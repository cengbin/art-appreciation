## Context

当前项目使用第三方库 `react-viewer` 实现图片预览功能。该库虽然功能完善，但存在以下问题：
- 依赖体积大（约 50-100KB）
- 定制化受限，难以深度定制UI和交互
- 样式覆盖维护成本高
- 缺少缩略图导航功能

现有代码位于 `src/pages/Images/index.tsx`，直接使用 `react-viewer` 的 `<Viewer>` 组件。

技术栈：React 18+ with TypeScript, SCSS for styling。

## Goals / Non-Goals

**Goals:**
- 创建功能完整的自定义图片预览组件，完全替换 react-viewer
- 实现核心功能：缩放、旋转、拖拽、导航、键盘快捷键
- 添加缩略图导航功能，支持快速浏览和跳转
- 实现懒加载和响应式设计，优化性能
- 减小包体积，移除第三方依赖
- 提供清晰的代码结构和 TypeScript 类型支持

**Non-Goals:**
- 不支持下载功能
- 不支持视频预览
- 不支持图片编辑功能
- 不实现虚拟滚动（可作为未来优化）

## Decisions

### 决策 1: 组件架构 - 自定义 Hooks 模式

**选择**: 使用自定义 Hooks 分离业务逻辑

**理由**:
- 逻辑复用：useImageTransform、useDrag、useKeyboard 可独立测试和复用
- 代码清晰：主组件只负责渲染，逻辑在 Hooks 中
- 易于维护：每个 Hook 职责单一

**替代方案**:
- Redux/Context: 过度设计，组件状态不需要全局共享
- Class Component: Hooks 更简洁，更符合 React 现代实践

### 决策 2: 导航按钮布局 - 左右两侧独立显示

**选择**: 导航按钮分别位于画面左右两侧，垂直居中

**理由**:
- 更直观：按钮位置与导航方向一致
- 更大的点击区域：不受底部容器限制
- 更好的视觉平衡：左右对称

**替代方案**:
- 底部居中容器：原 ViewerNavigation 实现，但视觉上不够直观
- 顶部导航：与关闭按钮冲突，不符合用户习惯

### 决策 3: 缩略图懒加载 - Intersection Observer API

**选择**: 使用 Intersection Observer 实现懒加载

**理由**:
- 性能优秀：浏览器原生支持，不占用主线程
- 简单易用：API 简洁，代码量少
- 兼容性好：现代浏览器全支持

**替代方案**:
- 滚动事件监听：需要手动计算位置，性能较差
- react-lazyload 库：增加依赖，功能过度

### 决策 4: 样式方案 - SCSS + CSS Variables

**选择**: 使用 SCSS 编写样式，CSS Variables 定义主题变量

**理由**:
- SCSS 嵌套：代码结构清晰，易于维护
- CSS Variables：运行时可修改，支持主题切换
- 响应式：媒体查询集中管理

**替代方案**:
- CSS-in-JS: 增加运行时开销，不符合项目现有技术栈
- Tailwind CSS: 需要重构现有样式系统

### 决策 5: 状态管理 - 本地 useState

**选择**: 使用 React useState 管理组件状态

**理由**:
- 状态局部化：只在 ImageViewer 组件内使用
- 简单直接：不需要复杂的状态管理
- 性能足够：状态更新不频繁

**替代方案**:
- Redux: 过度设计，增加复杂度
- Zustand: 不需要跨组件共享状态

### 决策 6: 缩略图尺寸 - 响应式自适应

**选择**: 根据屏幕宽度自动调整缩略图尺寸
- 桌面端（>768px）: 70px
- 平板端（480-768px）: 55px
- 手机端（<480px）: 50px

**理由**:
- 更好的用户体验：不同设备显示合适的尺寸
- 性能优化：小屏幕加载小图片
- 一致性：与其他响应式元素保持一致

**替代方案**:
- 固定尺寸：在小屏幕上显示效果差
- 用户可配置：增加复杂度，收益不大

## Risks / Trade-offs

### 风险 1: 浏览器兼容性
**风险**: Intersection Observer API 在旧浏览器不支持
**缓解**: 
- 目标浏览器为现代浏览器（Chrome/Firefox/Safari/Edge 最新版）
- 如需支持旧浏览器，可添加 polyfill

### 风险 2: 大量图片性能
**风险**: 100+ 张图片时可能出现性能问题
**缓解**:
- 懒加载减少初始渲染压力
- 未来可实现虚拟滚动优化
- 建议单次展示图片数量不超过 200 张

### 风险 3: 移动端手势支持
**风险**: 当前设计主要针对桌面端，移动端手势支持有限
**缓解**:
- 基础触摸拖拽已支持
- 未来可添加双指缩放等高级手势
- 响应式设计确保基本可用性

### 权衡 1: 功能完整性 vs 包体积
**权衡**: 自研组件功能可能不如 react-viewer 完善
**决策**: 优先实现核心功能，满足当前需求即可
**收益**: 减小包体积 50-100KB，提升加载速度

### 权衡 2: 开发时间 vs 定制化能力
**权衡**: 自研需要更多开发时间
**决策**: 投入 2-3 个工作日开发
**收益**: 完全可控，易于扩展和维护
